## 一、项目概述

### 1.1 目标
提供 Web 页面，用户输入公开 GitHub 仓库 URL，系统自动分析仓库数据并可视化展示相关指标，结合 AI 给出项目评分。

### 1.2 核心功能
1. **仓库信息提取** - 用户输入 GitHub URL，自动解析仓库信息
2. **基础指标展示** - Star 数、Fork 数、Watch 数、Branch 数、Issue 数、PR 数等
3. **编程语言分布** - 饼图/柱状图展示各语言占比
4. **贡献者统计** - Top 贡献者排行、贡献活跃度趋势
5. **项目健康度** - 最近更新频率、Issue 响应时间、PR 合并率
6. **AI 项目评分** - 结合多维度数据给出 0-100 综合评分及评语

## 三、GitHub API 使用方案

### 3.1 API 端点

| 功能 | Endpoint | 限流 |
|------|----------|------|
| 仓库信息 | `GET /repos/{owner}/{repo}` | 5000/小时 |
| 语言统计 | `GET /repos/{owner}/{repo}/languages` | 5000/小时 |
| Star/Fork | `GET /repos/{owner}/{repo}` (包含) | 5000/小时 |
| 贡献者 | `GET /repos/{owner}/{repo}/contributors` | 5000/小时 |
| 最近提交 | `GET /repos/{owner}/{repo}/commits` | 5000/小时 |
| Issue 列表 | `GET /repos/{owner}/{repo}/issues` | 5000/小时 |
| PR 列表 | `GET /repos/{owner}/{repo}/pulls` | 5000/小时 |
| README 文档 | `GET /repos/{owner}/{repo}/readme` | 5000/小时 |

### 3.2 认证方案
- **方案一**: 使用 GitHub Personal Access Token（简单，5000次/小时）
- **方案二**: 使用 GitHub App（更安全，适合生产环境）
- **推荐**: 方案一，开发阶段足够使用

### 3.3 限流应对策略
1. 添加缓存，已请求数据缓存 5-15 分钟
2. 批量获取，一次性拉取所有需要的数据
3. 错误重试，429 时自动等待后重试

## 五、AI 评分方案

### 5.1 评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 活跃度 | 25% | 最近提交频率、Issue/PR 处理速度 |
| 社区生态 | 25% | Star 数、贡献者数量、外部依赖 |
| 代码质量 | 20% | 文档完整性、CI/CD 配置、测试覆盖 |
| 维护状态 | 20% | 最近更新时间、维护响应速度 |
| 热度趋势 | 10% | Star 增长趋势、PR 数量 |

### 5.2 评分算法

```
总分 = Σ(维度分数 × 权重)

各维度分数计算:
- 活跃度 = min(提交频率得分 + Issue处理得分, 100)
- 社区生态 = min(Star得分 + 贡献者得分, 100)
- 代码质量 = 文档得分 × 0.6 + CI/CD得分 × 0.4
- 维护状态 = 更新及时性得分 × 0.5 + 响应速度得分 × 0.5
- 热度趋势 = Star增长指数 × 100
```

### 5.3 LLM 提示词设计

```json
{
  "role": "system",
  "content": "你是一位专业的开源项目分析师。请根据以下GitHub仓库数据，给出0-100的综合评分和详细评语。"
}
{
  "role": "user", 
  "content": "仓库名: {repo_name}\nStars: {stars}\nForks: {forks}\n语言分布: {languages}\n贡献者数: {contributors}\n最近更新: {last_updated}\nIssue数: {open_issues}\n描述: {description}\n\n请给出评分和评语，格式如下:\n评分: {score}/100\n评语: {comment}"
}
```
